## Architecture overview

8-module Maven service mirroring `blank-service` (RULE-004):

```
loyalty-ledger-domain/
  ├── loyalty-ledger-domain-core         # Balance, LedgerEntry, domain events
  └── loyalty-ledger-application-service # use cases, ports, outbox helper
loyalty-ledger-api                       # @RestController (vendor media type)
loyalty-ledger-data-access               # JPA entities, repos, outbox table
loyalty-ledger-message/
  ├── loyalty-ledger-message-core        # Kafka producer/consumer adapters
  └── loyalty-ledger-message-model       # Avro schema for points-credited
loyalty-ledger-container                 # @SpringBootApplication, application.yaml
loyalty-ledger-acceptance-test           # Cucumber + Testcontainers + Wiremock
loyalty-ledger-support                   # docker-compose for local infra
```

## Cross-cutting concerns

- **Topic creation:** `points-credited` is auto-created or pre-provisioned
  by Platform team before deploying.
- **Schema registry:** `PointsCreditedAvroModel` registered with
  `BACKWARD` compatibility from day 1.
- **Operational dashboards:** `outbox_status='STARTED'` row count alert
  (lag indicator) — owned by SRE.

## Risks

- **R1** — order-service's Avro schema evolves before we go live.
  Mitigation: pin to a specific commit of `order-service-message-model`;
  upgrade explicitly.
- **R2** — initial points-per-dollar rate is unconfirmed (open question
  in PRD). Mitigation: implement as a `loyalty-ledger.points-per-dollar`
  config key with a documented default.
